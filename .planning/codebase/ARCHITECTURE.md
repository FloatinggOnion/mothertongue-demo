# Architecture

**Analysis Date:** 2026-03-20

## Pattern Overview

**Overall:** Full-stack Next.js application using a Client-Server architecture with a React frontend consuming REST APIs, paired with server-side AI services integration via Google Gemini API for language understanding and generation.

**Key Characteristics:**
- Server Components for data fetching and AI logic (`'use server'`)
- Client Components for interactive UI and speech handling (`'use client'`)
- API route handlers in `/api` that bridge frontend and external AI services
- Modular component architecture with shared hooks
- Scenario-based conversation framework

## Layers

**Presentation Layer:**
- Purpose: Render UI and capture user input (speech/text)
- Location: `src/app/page.tsx`, `src/app/drill/[id]/page.tsx`, `src/components/`
- Contains: Page components, UI components (MessageBubble, MicButton, ReplySuggestions, FeedbackCard)
- Depends on: Hooks layer, Types layer, API routes
- Used by: Browser client

**API Layer:**
- Purpose: Handle HTTP requests, orchestrate AI service calls, translate between client and backend
- Location: `src/app/api/chat/route.ts`, `src/app/api/evaluate/route.ts`, `src/app/api/transcribe/route.ts`, `src/app/api/tts/route.ts`, `src/app/api/suggestions/route.ts`
- Contains: POST route handlers using Next.js server functions
- Depends on: Services layer, Config layer, Types layer
- Used by: Frontend via fetch() calls

**Services Layer:**
- Purpose: Encapsulate business logic and external service calls
- Location: `src/services/gemini.ts`
- Contains: Functions for Gemini API interaction (conversation, evaluation, suggestions, translation)
- Depends on: Types layer, Google Gemini SDK
- Used by: API route handlers

**Configuration Layer:**
- Purpose: Static scenario definitions and game content
- Location: `src/config/scenarios.ts`
- Contains: Scenario objects defining character, context, and difficulty level
- Depends on: Types layer
- Used by: Pages, API routes, Services

**Types Layer:**
- Purpose: TypeScript interfaces and type definitions
- Location: `src/types/index.ts`
- Contains: Core domain types (Scenario, Message, Evaluation, DrillSession, etc.)
- Depends on: None (leaf layer)
- Used by: All layers

**Hooks Layer:**
- Purpose: Encapsulate client-side stateful logic and browser APIs
- Location: `src/hooks/useSpeech.ts`
- Contains: useSpeechRecognition (browser/fallback), useSpeechSynthesis (TTS client)
- Depends on: Browser APIs (Web Speech API, MediaRecorder), fetch API
- Used by: Client components

## Data Flow

**Conversation Flow:**

1. **User speaks into microphone** → useSpeechRecognition hook captures audio via Web Speech API or MediaRecorder fallback
2. **Audio uploaded to /api/transcribe** → Transcribed by Gemini 2.0-flash with `audio/webm` MIME type
3. **Transcription returned to client** → Rendered in ConversationView and sent to /api/chat
4. **POST to /api/chat with conversation history** → getPartnerResponse() called with scenario context + proficiency level
5. **Gemini generates AI response** → System prompt conditions behavior (beginner/intermediate/advanced)
6. **Response returned with translation** → useSpeechSynthesis fetches MP3 from /api/tts
7. **Google Cloud Text-to-Speech generates audio** → Audio played to user via audio element
8. **User idle for 10 seconds** → Silent timer triggers /api/suggestions for reply hints
9. **User taps "End Drill"** → /api/evaluate called with full conversation history
10. **Silent evaluator (Gemini) scores performance** → Evaluation returned and FeedbackCard displayed

**State Management:**
- Client-side: React useState in drill page (`messages`, `isLoading`, `showSuggestions`, `evaluation`, etc.)
- Server-side: None - stateless API handlers
- Persistence: None - drills exist only during session (not saved)

## Key Abstractions

**Scenario:**
- Purpose: Define a conversational context (character, setting, starting prompt, difficulty)
- Examples: `src/config/scenarios.ts` exports 5 scenarios (market-haggling, greeting-elders, ordering-food, chatting-friends, transportation)
- Pattern: Static configuration objects with id, title, context, aiRole, difficulty, gender

**Message:**
- Purpose: Represent a single utterance in conversation
- Contains: id, role ('user'|'ai'), content, optional translation, timestamp
- Used in: Conversation history tracking, rendering MessageBubbles

**Evaluation:**
- Purpose: Structured feedback on user's language performance
- Contains: strength, improvement, correctedSentence, scores (overall, fluency, grammar, confidence)
- Generated by: callGemini with evaluatorSystemPrompt (silent evaluation)

**GeminiResponse:**
- Purpose: Standardized response format from chat endpoint
- Contains: reply (AI message), translation (English version), suggestions (ReplySuggestion[]), shouldEndDrill flag
- Generated by: getPartnerResponse() function

## Entry Points

**Home Page:**
- Location: `src/app/page.tsx`
- Triggers: User navigates to `/`
- Responsibilities: Display hero section, list available scenarios, navigate to drill via ScenarioCard

**Drill Page:**
- Location: `src/app/drill/[id]/page.tsx` (dynamic route)
- Triggers: User clicks on ScenarioCard or navigates to `/drill/[scenarioId]`
- Responsibilities:
  - Initialize conversation with scenario starter prompt
  - Manage microphone/text input and message history
  - Fetch AI responses via /api/chat
  - Display conversation in ConversationView
  - Fetch suggestions on silence via /api/suggestions
  - Trigger evaluation via /api/evaluate on drill end

**API Routes:**
- `POST /api/chat` - Generate AI partner response
- `POST /api/evaluate` - Evaluate conversation and return structured feedback
- `POST /api/transcribe` - Convert audio blob to text via Gemini
- `POST /api/tts` - Convert text to speech MP3 via Google Cloud TTS
- `POST /api/suggestions` - Generate 2 reply hints based on AI's last message

## Error Handling

**Strategy:** Try-catch with fallback values and user-friendly error messages

**Patterns:**
- API routes wrap main logic in try-catch, return 500 with error message on failure
- Services layer (gemini.ts) includes rate-limit retry logic (429 → wait 8s → retry once)
- Speech recognition falls back to MediaRecorder + Gemini transcription if native API fails
- Suggestions endpoint returns hardcoded fallback suggestions if Gemini call fails
- Evaluation returns default 5-score evaluation if JSON parsing fails
- Missing scenarios trigger 404 responses from API routes
- Text-to-Speech client handles auth errors with specific messaging about credentials

## Cross-Cutting Concerns

**Logging:**
- Uses console.error/console.log for debugging
- No centralized logging framework
- Found in: error handlers, speech recognition fallback detection, rate-limit detection

**Validation:**
- API routes validate required fields (text, audio, scenarioId)
- return 400 Bad Request if missing
- Scenario lookup validates scenarioId exists before processing
- Client-side: disabled submit buttons until input is valid

**Authentication:**
- None for user actions (public app)
- API keys managed via environment variables:
  - GEMINI_API_KEY for Google Gemini API
  - GOOGLE_CLIENT_EMAIL and GOOGLE_PRIVATE_KEY for Google Cloud TTS (credentials object)
- No user session management

**Proficiency Level:**
- Hardcoded to 'beginner' in drill page (line 30)
- Passed to getPartnerResponse and getReplySuggestions to adjust AI behavior
- Affects system prompt conditioning: beginner uses more English + encouragement, advanced uses primarily Yoruba

**Gender:**
- Stored in Scenario object (male/female)
- Passed to useSpeechSynthesis.speak() to select voice
- Drives Google Cloud TTS voice mapping: male='en-NG-Wavenet-B', female='en-NG-Wavenet-A'
